AI Request Orchestration

// browser.js · Engineering Design Spec

Goal & Scope

When the user sends a message, the assistant must decide — before producing any visible output — which tools and context pieces it actually needs. Running everything unconditionally is expensive and slow for a 2.5 GB local model. The pipeline described here composes a chain of lightweight decisions that progressively load only the context required.

Primary question: Can I answer this from general knowledge, or do I need the current page? And if I need the page, do I also need to act on it?

Non-goals (v1)

  • No free-form JS execution — only the explicit action dictionary.
  • No multi-tab awareness — always operates on the active tab.
  • No model fine-tuning or LoRA — prompt engineering only.

High-Level Pipeline

1
Request Intake

Receive message, attach tab metadata, short conversation context, memory snippets. Start request_id.

2
M1 · Scope Classifier

One LLM call: general or page_related? Also fires memory lookup and conversation extraction concurrently.

3
General branch → M7 Response Generator

No HTML work. Straight answer. Done.

4
M2 · HTML Simplifier

Convert DOM to semantic Markdown, append a compact visible element map, and return size metrics for budget control.

5
M3 · Context Assembler + M4 · Info Gap Checker

Merge memory, conversation facts, HTML artifact. Check if anything is still missing. Loop with follow-up question if needed.

6
M5 · Budget Enforcer

Token count assembled context. If over budget: aggressive reduction → relevance selector pass → snippets only.

7
M6 · Action Planner

Does the request require browser actions? Map to action dictionary, validate, execute, capture result.

8
M7 · Response Generator

Final answer assembled from all prior outputs. Optionally includes action trace.

M1 · Scope Classifier

The cheapest inference call in the pipeline. Runs immediately after intake, in parallel with memory lookup and conversation extraction.

🔍

classifyScope(request, pageMetadata)

Output: scope + confidence + rationale
  • User message
  • Page URL + title
  • Last 2 conversation turns
  • scope: "general" | "page_related"
  • confidence: 0.0–1.0
  • rationale: string
📝 Prompt — M1 Scope Classifier
You are a request router for a browser assistant.
Classify the user's request into one of two scopes:

- "general": answerable from general knowledge alone, no need to see the page.
- "page_related": requires understanding the current page's content, structure, or state.

Page context (for reference only, do not analyse content):
URL: {{url}}
Title: {{title}}

Recent conversation (last 2 turns):
{{conversation_snippet}}

User request: "{{user_message}}"

Respond with a JSON object only, no explanation:
{
  "scope": "general" | "page_related",
  "confidence": 0.0-1.0,
  "rationale": "one sentence"
}

Signals for page_related:
- Words like "this", "here", "this button", "this section", "on this page"
- Accessibility or layout questions about current content
- Requests to interact with, fill, click, or modify elements
- Questions whose answer depends on visible DOM content

Signals for general:
- Conceptual or standards questions independent of any page
- Coding help, explanations, comparisons not referencing current content
- The page URL/title is clearly irrelevant to the question

Confidence threshold policy

If confidence < 0.65 and scope is general, treat as page_related to avoid missing context. Log the ambiguous case for analytics. This is conservative by design — HTML acquisition is cheap relative to a wrong answer.

Skip M2–M6. Go directly to M7 with message + memory facts only.

Continue to M2. Also fire M3 context assembly in parallel (memory + conversation — no HTML yet).

M2 · HTML Simplifier

Builds an AI-friendly page snapshot from the live DOM. This is a JS pipeline running in the renderer process (no LLM). The result is semantic Markdown plus a visible element map, then fed to M3/M5/M6.

⚙️

simplifyCurrentPage(webContents, currentUrl) → PageSnapshot

Pure JS pipeline via executeJavaScript; optional cache is upstream.

Simplification pipeline (actual implementation order)

  1. Load dom-to-semantic-markdown/dist/browser/bundle.js once and cache the bundle content in main-process memory.
  2. In the page context, read document.documentElement.outerHTML; if empty, return empty markdown.
  3. Convert HTML to Markdown with: extractMainContent: true, refifyUrls: true, includeMetaData: "basic", websiteDomain: origin(currentUrl).
  4. Build a visible element map from interactive/semantic selectors: a, button, input, select, textarea, label, form, [role], [aria-label], [name], [id], [class].
  5. For each candidate: keep only visible elements (display/visibility/opacity + non-zero bounding rect), then emit one compact line: tag#id.class1.class2 name="" type="" role="" aria-label="" href="" text="".
  6. Apply hard caps to bound size: MAX_ELEMENTS=350, MAX_CLASSES=4, MAX_TEXT_LENGTH=70, deduplicate identical lines.
  7. Append map as ## Visible Element Map under the semantic markdown and return metrics: markdownLength, elementMapLength, elementCount.
  • markdown: string (semantic markdown + optional element map)
  • markdownLength: number
  • elementMapLength: number
  • elementCount: number
  • Throws if final markdown is empty
  • Element map is visibility-aware and deduplicated
  • No explicit DOM fingerprinting inside M2
  • Caching strategy is handled outside this module
⚠ Do not pass raw outerHTML to the model. M2 already emits a compressed snapshot, but large pages can still exceed budget. M5 must trim snapshot sections deterministically before M7.

M3 · Context Assembler

Collects all non-HTML context: memory snippets, conversation facts (excluding raw HTML messages), and user-provided follow-up answers. Runs partially in parallel with M2.

🗂️

assembleContext(request) → ContextBundle

Sub-steps (run in parallel where possible)

  1. Memory lookup — vector or keyword search over stored user facts. Return top-3 relevant snippets. Each snippet: { source: "memory", key, content, relevance_score }.
  2. Conversation extraction — scan last N turns (exclude any turn containing raw HTML). Extract factual statements relevant to the current request using the prompt below. Return deduplicated list.
  3. Follow-up answers — if M4 previously triggered a follow-up, attach the stored user answer as a { source: "user_clarification", content } fact.
📝 Prompt — Conversation Fact Extractor
You are extracting factual information from a conversation history.
Goal: find facts that are relevant to answering the user's current request.
Ignore any turns that contain raw HTML markup.

Current user request: "{{user_message}}"

Conversation history (oldest first):
{{conversation_history}}

Return a JSON array of extracted facts. Each fact:
{ "content": "short factual statement", "turn_index": N }

Return empty array [] if nothing is relevant.
No explanation, JSON only.

M4 · Info Gap Checker

After M3 assembles context (and M2 provides the HTML artifact), M4 asks: is there still something missing that would make the answer unreliable? This runs as an LLM call with the assembled context but without the HTML (to keep it cheap).

❓

checkInfoGap(request, contextBundle) → GapResult

📝 Prompt — M4 Info Gap Checker
You are deciding whether you have enough information to answer a user's request reliably.

User request: "{{user_message}}"

Available context facts:
{{context_facts_list}}

Page: {{url}} — {{title}}

Answer with a JSON object only:
{
  "has_enough_info": true | false,
  "missing": "brief description of what is missing, or null",
  "followup_question": "one focused question to ask the user, or null"
}

Rules:
- If the request is clear and the available facts are sufficient → has_enough_info: true
- Only request missing information that is (a) genuinely needed and (b) the user is likely to know
- Ask at most ONE question. Make it specific and short.
- Do not ask about page content — the page HTML will be provided separately.

Loop behaviour

If has_enough_info = false: surface followup_question to the user. Wait for answer. Store answer in state as a fact. Re-run M4 once. If still missing after one retry → proceed anyway, flag uncertainty in M7.

Continue to M5.

Emit follow-up question to UI. On user response, store and loop once. Then continue to M5 regardless.

M5 · Budget Enforcer

Ensures the full prompt sent to M7 stays within the model's context window. The model is local (~2.5 GB) — assume 4096 tokens total budget as a conservative default (adjust for your specific model).

📐

enforceContextBudget(state) → FinalContext

Budget allocation (tokens)

// Approximate allocation for a 4096-token budget
{
  system_prompt:   300,   // fixed assistant persona + rules
  user_message:    150,   // the actual request
  context_facts:   300,   // memory + conversation extracts
  action_schema:   200,   // action dictionary excerpt (if M6 detected actions)
  html_budget:     2800,  // remaining for HTML
  response_reserve:346,  // max_new_tokens reserved
}

Reduction cascade

  1. Estimate tokens from the full snapshot (semantic markdown + visible element map). If within html_budget → pass as-is.
  2. If over budget: trim the element map first (keep high-signal lines containing id, name, aria-label, role, links, and controls).
  3. If still over budget: run Relevance Selector over chunked markdown sections and keep returned section IDs only.
  4. If still too large: pass headings + top-ranked sections + a compact control list summary.
📝 Prompt — M5 Relevance Selector
The following page is too long to include in full. 
Select the sections most relevant to answering the user's request.

User request: "{{user_message}}"

Page snapshot sections (each has an id):
{{sectioned_snapshot}}

Return a JSON array of section ids to keep, most important first:
["section-id-1", "section-id-2", ...]

Keep only what is truly needed. Prefer: forms, main content, visible controls, links/buttons
referenced by the request. Omit: footers, cookie banners, ads, sidebars
unless directly referenced.

M6 · Action Planner

Determines whether the request requires performing browser operations (clicking, filling, highlighting, etc.) and maps the intent to the safe action dictionary.

⚡

planActions(request, pageSnapshot) → ActionPlan

📝 Prompt — M6 Action Planner
You are a browser action planner. 
Decide if the user's request requires performing actions on the page, 
and if so, produce a step-by-step action plan using only the allowed actions.

User request: "{{user_message}}"

Page snapshot (semantic markdown + visible element map):
{{page_snapshot}}

Allowed actions (use these names exactly):
{{action_dictionary_schema}}

Respond with JSON only:
{
  "action_required": true | false,
  "plan": [
    {
      "step": 1,
      "action": "action_name",
      "args": { ... },
      "rationale": "why this step"
    }
  ]
}

Rules:
- If no action is needed (explanation/analysis only) → action_required: false, plan: []
- Use only actions from the allowed list. Do not invent new ones.
- Prefer element references coming from the visible element map (id/name/aria-label/role/text).
- Keep plans minimal. Do not add steps "just in case".

Execution

Each step in the plan is validated against the action dictionary schema before execution. Argument types and selector safety are checked. On execution error, the step is skipped and its error is recorded in action_plan[i].result. A rollback is attempted if available.

M7 · Response Generator

The final LLM call. This is the only call the user sees. All previous modules are invisible bookkeeping. Construct the prompt carefully — it is the most expensive call.

💬

generateResponse(state) → FinalResponse

📝 System Prompt — M7 Response Generator (base)
You are a helpful browser assistant embedded in a web browser.
You have access to the current page content and can perform safe browser actions.
Be concise. Prefer short answers unless depth is clearly needed.
Never reveal your internal reasoning steps or tool names.
If you performed actions on the page, mention what you did briefly at the end.
📝 User Turn — M7 (page_related, with actions)
[CONTEXT FACTS]
{{context_facts}}

[PAGE: {{url}} — {{title}}]
{{page_snapshot_or_snippets}}

[ACTIONS PERFORMED]
{{action_results}}
(empty if no actions were taken)

[USER REQUEST]
{{user_message}}

Answer the user's request. 
If information is uncertain due to incomplete page context, say so briefly.
If actions were performed, confirm what was done.
📝 User Turn — M7 (general scope)
[CONTEXT FACTS]
{{context_facts}}

[USER REQUEST]
{{user_message}}

Action Dictionary (v1)

All browser operations are enumerated here. M6 may only reference these by name. No free-form JS execution.

highlight(selector) Add a visible highlight outline to a matched element. safe
scrollTo(selector) Smooth-scroll the page so the element is visible. safe
focusElement(selector) Move browser focus to an interactive element. safe
fillInput(selector, value) Set the value of a text input or textarea. confirm
clickElement(selector) Simulate a click on a button, link, or checkbox. confirm
selectOption(selector, value) Select an option in a <select> element. confirm
injectCSS(css) Temporarily inject CSS for previewing style changes. Reverted on next navigation. confirm
removeHighlights() Remove all assistant-added highlights. safe
copyToClipboard(text) Write a string to the system clipboard. safe

safe = execute immediately. confirm = show confirmation UI before executing if the user did not explicitly request the action.

Action schema (M6 receives this)

[
  { "name": "highlight",       "args": { "selector": "string (CSS selector)" } },
  { "name": "scrollTo",        "args": { "selector": "string" } },
  { "name": "focusElement",    "args": { "selector": "string" } },
  { "name": "fillInput",       "args": { "selector": "string", "value": "string" } },
  { "name": "clickElement",    "args": { "selector": "string" } },
  { "name": "selectOption",    "args": { "selector": "string", "value": "string" } },
  { "name": "injectCSS",       "args": { "css": "string (raw CSS)" } },
  { "name": "removeHighlights", "args": {} },
  { "name": "copyToClipboard",  "args": { "text": "string" } }
]

All Prompts — Summary

MODULE PURPOSE EST. TOKENS OUTPUT FORMAT
M1 Scope general vs page_related ~200 JSON
M3 Conv. Extract pull facts from history ~300 JSON array
M4 Gap Check detect missing info ~400 JSON
M5 Relevance select snapshot sections ~600 JSON array of ids
M6 Action Plan produce action steps ~800 JSON

Total inference calls per request: 1 (general) or 2–5 (page_related, depending on budget and follow-up). M1, M3, M4, M6 are cheap classifiers; M7 carries the cost.

Concurrency Model

// Pseudocode showing parallel vs sequential steps

async function handleRequest(request) {

  // ── PHASE 1: always-parallel ──────────────────────────
  const [scope, memoryFacts, convFacts, cachedSnapshot] = await Promise.all([
    M1_classifyScope(request),          // LLM call
    lookupMemory(request.message),       // vector search
    extractConvFacts(request.message),   // LLM call
    getPageCache(request.tab_id),        // cache read (optional snapshot cache)
  ]);

  if (scope === "general") {
    return M7_respond({ request, memoryFacts, convFacts });
  }

  // ── PHASE 2: page snapshot acquisition (sequential) ───
  const pageSnapshot = cachedSnapshot ?? await M2_simplifyCurrentPage(request.tab_id, request.url);

  // ── PHASE 3: gap check + budget in parallel ───────────
  const contextBundle = M3_assembleContext({ memoryFacts, convFacts });
  const gap = await M4_checkGap(request, contextBundle);

  if (!gap.has_enough_info) {
    const answer = await askUser(gap.followup_question);
    contextBundle.facts.push({ source: "user_clarification", content: answer });
    // re-check once, then continue regardless
  }

  // ── PHASE 4: budget + actions in parallel ─────────────
  const [finalContext, actionPlan] = await Promise.all([
    M5_enforceBudget({ pageSnapshot, contextBundle, request }),
    M6_planActions(request, pageSnapshot),
  ]);

  const actionResults = await executeActionPlan(actionPlan);

  // ── PHASE 5: final response ───────────────────────────
  return M7_respond({ request, finalContext, actionResults });
}

Open Questions

Confidence threshold (M1)

Current proposal: 0.65. Below that, default to page_related. Track false-positive rate in analytics to tune over time. A too-low threshold wastes M2 work; too high produces wrong-scope misses.

Token budget

The charCount / 4 estimate is fast but inaccurate for CJK text or code-heavy pages. Consider running the model's actual tokenizer (if accessible from Node) as an optional accurate pass for pages near the budget boundary (±15%).

Follow-up: single vs multi-question

Keep it single-question for v1. Multi-question forms feel interrogative and slow. If the single question doesn't resolve the gap, proceed with uncertainty flagged in M7's response.

Page cache TTL

30 seconds works for most SPAs. Consider listening to MutationObserver for significant DOM changes and invalidating the cache proactively rather than relying purely on TTL.

v1 action set

The 9 actions above cover the most common browser assistant tasks. Defer navigateTo(url) and submitForm(selector) to v2 — they have higher blast radius and need stricter confirmation flows.

Model context limit

The 4096-token budget assumes a mid-range local model. If your model supports 8k or 16k context, adjust the html_budget allocation proportionally. The module structure remains identical.
